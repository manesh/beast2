**Beast2 Media Library & Database Architecture**  
  
**Overview**  
  
The Beast2 media system manages all locally generated and imported assets.  
  
Supported asset types include:  
	•	images  
	•	videos  
	•	latents  
	•	prompts  
	•	generators  
	•	knowledge databases  
  
The library must efficiently scale to:  
	•	**100k–1M media files**  
	•	**10TB+ storage**  
	•	**tens of thousands of tags**  
  
Primary asset examples include:  
	•	videos generated by ++Wan 2.2++  
	•	images generated by ++Stable Diffusion++  
	•	images generated by ++SDXL++  
	•	videos generated by ++LTX Video++  
  
The system also supports **structured text databases used by LLM workflows**.  
  
⸻  
  
**Current implementation note**  
  
The current repository implementation provides the first working slice of the
media library described in this document, including:
	•	filesystem media persistence  
	•	SQLite metadata indexing  
	•	tag and generator metadata tables  
	•	generator history records  
	•	thumbnail sidecars stored under `thumbs/`  
	•	latent records stored in the `latents` table  
	•	named local knowledge sources and entries stored in SQLite  
  
At the moment, this is a **native local media layer** intended to support the
current execution pipeline. Advanced browsing queries, richer preview formats,
UI gallery loading, and large-scale import workflows are still future
expansions.  
  
⸻  
  
**Design Goals**  
  
**Massive local libraries**  
  
The system must handle:  
  
```
100k–1M files

```
  
Without UI slowdowns.  
  
⸻  
  
**Fast browsing**  
  
The gallery must support:  
  
```
thousands of preview thumbnails
simultaneous video loops
tag filtering

```
  
without blocking generation tasks.  
  
⸻  
  
**Spinning disk compatibility**  
  
Many users will use large HDD arrays.  
  
Typical workstation setup:  
  
```
10TB–40TB spinning storage

```
  
Therefore:  
  
```
sequential reads preferred
metadata stored in SSD database

```
  
  
⸻  
  
**Offline-first**  
  
Everything must function with:  
  
```
no internet
no cloud services

```
  
  
⸻  
  
**Flexible database workflows**  
  
The system must support **LLM-based knowledge workflows**, such as:  
	•	local encyclopedia  
	•	belief databases  
	•	structured knowledge bases  
  
⸻  
  
**Media Types**  
  
Beast2 supports several media types.  
  
**Image**  
  
Example:  
  
```
webp
png
jpg

```
  
  
⸻  
  
**Video**  
  
Example:  
  
```
webm
mp4

```
  
Default generator output is usually:  
  
```
webm

```
  
  
⸻  
  
**Latent**  
  
Binary tensor files representing model latent states.  
  
Example:  
  
```
latent tensors
motion vectors
explorer anchors

```
  
  
⸻  
  
**Generator**  
  
Beast2 generator specification files.  
  
Example:  
  
```
b2 generator specs
workflow definitions

```
  
  
⸻  
  
**Database**  
  
Structured text or vector databases used by generators.  
  
Example uses:  
  
```
encyclopedia
belief systems
knowledge bases
tag prediction models

```
  
  
⸻  
  
**File Storage Layout**  
  
Files are stored directly on disk.  
  
Example structure:  
  
```
/b2/
   media/
      images/
      videos/
      latents/
   generators/
   db/

```
  
Example:  
  
```
/b2/media/videos/wan22/

```
  
  
⸻  
  
**Database Layer**  
  
Metadata is stored in a dedicated database.  
  
This database stores:  
  
```
file metadata
tags
relationships
latent vectors
generator history

```
  
Media files themselves remain on disk.  
  
⸻  
  
**Database Engine**  
  
The recommended backend is:  
  
```
SQLite

```
  
Advantages:  
  
```
zero configuration
fast local queries
portable

```
  
SQLite can easily handle:  
  
```
millions of rows

```
  
when properly indexed.  
  
⸻  
  
**Database Tables**  
  
**media table**  
  
Stores metadata about each asset.  
  
Example fields:  
  
```
media_id
path
type
resolution
duration
size
creation_time
generator_id

```
  
  
⸻  
  
**tags table**  
  
Stores tag definitions.  
  
Example:  
  
```
tag_id
tag_name
tag_type

```
  
  
⸻  
  
**media_tags table**  
  
Associates tags with media.  
  
Example:  
  
```
media_id
tag_id
confidence

```
  
  
⸻  
  
**latents table**  
  
Stores metadata about latent files.  
  
Example:  
  
```
latent_id
media_id
latent_type
model
shape
file_path

```
  
  
⸻  
  
**generators table**  
  
Stores generator metadata.  
  
Example:  
  
```
generator_id
name
description
creation_date

```
  
  
⸻  
  
**Tagging System**  
  
Tags are central to browsing and discovery.  
  
Example tags:  
  
```
character
pixel-art
cinematic
smooth-motion
elf
cowboy

```
  
Tag counts may exceed:  
  
50,000+  
  
  
⸻  
  
**Tag Indexing**  
  
To maintain performance, the database includes indexes for:  
  
```
tag lookup
media lookup
tag intersection

```
  
Example index:  
  
```
index media_tags(tag_id)

```
  
This allows fast queries such as:  
  
```
videos tagged elf + pixel-art

```
  
  
⸻  
  
**Gallery Performance**  
  
The UI gallery loads previews using optimized queries.  
  
Example workflow:  
  
```
search query
↓
database returns media IDs
↓
UI loads preview files

```
  
Preview media should be stored as:  
  
```
small webp thumbnails
low bitrate video previews

```
  
This allows rapid browsing.  
  
⸻  
  
**Video Preview System**  
  
Videos automatically generate preview loops.  
  
Example:  
  
2–3 second low resolution preview  
  
These previews are cached for gallery playback.  
  
⸻  
  
**Latent Library**  
  
Latents are stored as binary files.  
  
Example storage:  
  
```
/b2/media/latents/

```
  
Typical sizes:  
  
```
image latent: ~50 KB

```
video latent: ~1–2 MB  
  
Libraries may contain:  
  
```
tens of thousands of latents

```
  
  
⸻  
  
**Knowledge Databases**  
  
Beast2 also supports **text databases used by generators**.  
  
These databases allow LLM workflows such as:  
  
```
local encyclopedia
belief databases
knowledge augmentation

```
  
  
⸻  
  
**Local Encyclopedia Workflow**  
  
Example generator workflow:  
  
```
user prompt
↓
extract search terms
↓
query local encyclopedia
↓
retrieve article
↓
feed context to LLM
↓
generate response

```
  
This allows powerful offline knowledge systems.  
  
⸻  
  
**Belief Databases**  
  
Belief databases allow LLM responses to reflect predefined viewpoints.  
  
Example database:  
  
```
b2_belief_db

```
  
Example contents:  
  
```
political beliefs
fictional world rules
character viewpoints
company policies

```
  
Workflow example:  
  
```
prompt
↓
extract keywords
↓
query belief_db
↓
inject context
↓
LLM response conditioned by beliefs

```
  
  
⸻  
  
**Database Media Type**  
  
Generators can treat databases as input media.  
  
Example:  
  
```
b2_input b2_database

```
  
Possible uses include:  
  
```
context lookup
knowledge injection
structured memory

```
  
  
⸻  
  
**LLM Output Databases**  
  
LLM workflows may also write data back into databases.  
  
Example:  
  
```
run prompt
↓
extract structured output
↓
insert into database

```
  
Example uses:  
  
```
knowledge extraction
tag prediction
script generation
prompt libraries

```
  
  
⸻  
  
**Database Indexing for Knowledge**  
  
Knowledge databases may contain:  
  
```
millions of text fragments

```
  
To support semantic retrieval, optional vector search may be used.  
  
Possible engines:  
  
```
sqlite vector extension
faiss

```
  
These enable:  
  
```
semantic search
context retrieval
memory systems

```
  
  
⸻  
  
**Disk Performance Optimization**  
  
Because spinning disks may be used, the system must minimize random I/O.  
  
Strategies include:  
  
```
metadata on SSD
media on HDD
sequential reads preferred

```
  
  
⸻  
  
**Background Indexing**  
  
When new media is created:  
  
```
generation complete
↓
metadata recorded
↓
thumbnail generated
↓
tags indexed

```
  
These tasks run in background threads.  
  
⸻  
  
**Logging Integration**  
  
Every generation produces log records.  
  
Logged information includes:  
  
```
generator used
model used
parameters
generation time

```
  
This allows full reproducibility.  
  
⸻  
  
**Scalability Limits**  
  
The architecture supports:  
  
```
1M+ media assets
100k+ tags
10TB+ storage

```
  
with minimal performance degradation.  
  
⸻  
  
**Summary**  
  
The Beast2 Media Library and Database System provides a scalable foundation for large local generative workflows.  
  
Key capabilities include:  
	•	efficient handling of massive media libraries  
	•	fast tag-based search  
	•	latent storage and indexing  
	•	LLM knowledge databases  
	•	belief conditioning systems  
	•	local encyclopedia workflows  
  
This design supports Beast2’s goal of functioning as a **complete offline generative AI workstation**.
