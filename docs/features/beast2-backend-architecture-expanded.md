**Beast2 Backend Architecture (Expanded)**  
  
Beast2 is a **local generative AI runtime and media workstation** designed to efficiently manage:  
	•	AI generation  
	•	latent libraries  
	•	large media collections  
	•	interactive exploration  
  
The system is optimized to handle:  
  
```
10k–100k media files
10k–100k tags
thousands of generation jobs
continuous GPU workloads

```
  
All while maintaining **smooth UI performance**.  
  
⸻  
  
**High-Level System Architecture**  
  
```
Beast2 Desktop App
      │
UI Layer
      │
Beast2 Core Runtime (C)
      │
──────────────────────────
Subsystems
──────────────────────────

GPU Scheduler
Generation Engine
Latent Library
Media Database
Filesystem Manager
Telemetry + Logging
Model Runtime Layer

```
  
Each subsystem is designed to minimize latency and contention.  
  
⸻

**Media Storage Strategy**  
  
Large collections require balancing filesystem and database responsibilities.  
  
**Key rule**  
  
```
media in filesystem
metadata in database

```
  
Do NOT store images/videos inside a database.  
  
This prevents:  
	•	database bloat  
	•	slow I/O  
	•	memory pressure  
  
⸻  
  
**Filesystem Strategy**  
  
To avoid directory slowdown with thousands of files, Beast2 uses **sharded directories**.  
  
Example:  
  
```
outputs/images/

ab/
ac/
ad/

```
  
Generated by hashing file ID.  
  
Example path:  
  
```
outputs/images/ab/ab39294.webp

```
  
Advantages:  
	•	avoids huge directory lists  
	•	faster file lookup  
	•	scalable to millions of files  
  
⸻  
  
**Media Database**  
  
Beast2 requires a fast local database.  
  
Recommended:  
  
**SQLite**  
  
Reasons:  
	•	embedded  
	•	extremely fast  
	•	zero server  
	•	mature indexing  
  
Database file:  
  
```
db/beast2.sqlite

```

**Indexing Strategy**  
  
Indexes are critical for performance.  
  
Recommended indexes:  
  
```
media(created_time)

media(model)

tags(name)

media_tags(tag_id)

media_tags(media_id)

```
  
This allows fast queries like:  
  
show videos tagged "walk_cycle"  
  
Even with **tens of thousands of files**.  
  
⸻  
  
**Thumbnail System**  
  
The UI should never load full media files.  
  
Instead Beast2 generates thumbnails.  
  
```
thumbs/
    ab/
    cd/

```
  
Formats:  
  
```
webp
mp4 preview

```
  
Benefits:  
	•	fast gallery loading  
	•	lower VRAM usage  
	•	minimal decoding overhead  
  
⸻  
  
**UI Performance Strategy**  
  
UI responsiveness must remain smooth even when the GPU is fully utilized.  
  
Key techniques:  
  
**Separate UI thread**  
  
UI must never block on:  
	•	database queries  
	•	generation jobs  
	•	video encoding  
  
⸻  
  
**Async database queries**  
  
All queries run through an async queue.  
  
```
UI request
   ↓
DB worker thread
   ↓
results returned

```
  
  
⸻  
  
**Virtualized gallery**  
  
The gallery only renders visible items.  
  
Example:  
  
```
10000 items total

```
30 visible  
  
Only those 30 load thumbnails.  
  
⸻  
  
**Video Playback Strategy**  
  
Playing looping videos in the gallery can stress GPU memory.  
  
Options:  
  
**CPU decoding**  
  
Safer but higher CPU load.  
  
**GPU decoding**  
  
Using hardware decode via FFmpeg.  
  
Recommended approach:  
  
```
GPU decode
CPU display

```
  
Frames are decoded on GPU but displayed via UI textures.  
  
VRAM usage must be capped.  
  
Example:  
  
max gallery video VRAM: 512 MB  
  
Old videos unload automatically.  
  
⸻  
  
**GPU Scheduler (Extended)**  
  
The scheduler balances:  
  
```
generation
latent previews
video decode
model inference

```
  
Job priorities:  
  
```
1. UI previews
2. interactive generation
3. batch generation

```
4. background tagging  
  
This ensures browsing the gallery never freezes the system.  
  
⸻  
  
**VRAM Allocation Strategy**  
  
Scheduler tracks:  
  
```
total VRAM
model VRAM
job VRAM
UI VRAM

```
  
Example allocation:  
  
```
24GB GPU

8GB models
12GB generation
2GB UI previews
2GB buffer

```
  
This prevents generation jobs from starving the UI.  
  
⸻  
  
**Latent Storage**  
  
Latents are stored as binary tensors.  
  
Example:  
  
```
latents/motion/walk_01.lat

```
  
Metadata stored in DB.  
  
Latents can be:  
  
```
tagged
grouped
interpolated

```
  
  
⸻  
  
**Tagging System**  
  
Tags enable organization and latent discovery.  
  
Example tags:  
  
```
walk_cycle
camera_pan
pixel_art
cinematic
character_concept

```
  
Users can:  
  
```
tag multiple files
tag latents
search by tag

```
  
  
⸻  
  
**Logging System**  
  
Logging is a core feature.  
  
Beast2 logs:  
  
```
generation time
GPU utilization
VRAM usage
queue length
job duration

```
  
Logs stored in:  
  
```
db/performance_logs

```
  
  
⸻  
  
**Performance Telemetry**  
  
To keep generation estimates accurate, Beast2 measures real-time performance.  
  
Metrics tracked:  
  
```
GPU utilization
queue wait time
UI GPU usage
generation step time

```
  
Example:  
  
```
wan22 generation
estimated: 30s
actual: 45s

```
  
The scheduler updates estimates dynamically.  
  
⸻  
  
**Performance Estimation Model**  
  
Estimates include:  
  
```
model type
resolution
steps
current GPU load
UI load

```
  
Example formula:  
  
```
generation_time =
baseline_model_time
× resolution_factor
× GPU_load_factor

```
  
If the user browses the gallery while generating, the system adjusts estimates.  
  
⸻  
  
**Job Queue Metrics**  
  
Each job tracks:  
  
```
queued_time
start_time
finish_time
gpu_time
cpu_time

```
  
This allows Beast2 to:  
	•	improve future estimates  
	•	detect bottlenecks  
	•	optimize scheduling  
  
⸻  
  
**Continuous Job Queue**  
  
Beast2 supports continuous generation.  
  
Example workflow:  
  
```
user browsing gallery
user queues new prompts
background jobs running

```
  
Scheduler dynamically adjusts job priorities.  
  
⸻  
  
**Model Cache**  
  
Models remain loaded in VRAM until needed elsewhere.  
  
Example cache state:  
  
```
SDXL loaded
Wan22 loaded
LLM loaded

```
  
Eviction policy:  
  
```
least recently used

```
