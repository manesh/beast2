**beast2-creation-template-subject-matter-expert-research-assistant**  
  
**Overview**  
  
The **Beast2 Subject Matter Expert Research Assistant** is a creation template that performs AI-assisted research directly on a local machine.  
  
It combines:  
	•	multi-engine search  
	•	automated content extraction  
	•	LLM interpretation  
	•	persistent knowledge storage  
  
The assistant assumes incoming queries are **related to Beast2 or concepts relevant to Beast2 design**.  
  
Instead of producing temporary answers, the system builds a **growing knowledge database** that becomes part of the Beast2 archive.  
  
Over time, this database evolves into a **shared encyclopedia of technical knowledge** useful for creators working with Beast2.  
  
⸻  
  
**Sales Headline**  
  
**Collaboratively building shareable encyclopedias which store valuable compressed artifacts of expensive compute.**  
  
⸻  
  
**Beast2 Philosophy**  
  
This creation reflects core Beast2 philosophy:  
  
**Compute once, reuse forever.**  
  
Every computation result becomes a reusable artifact.  
  
Nothing is wasted.  
  
Results are cached, stored, and shared so that future creators benefit from past work.  
  
Beast2 systems are designed to behave less like traditional software and more like **persistent intelligence engines**.  
  
⸻  
  
**First-Class Database Artifacts**  
  
This creation introduces an important Beast2 capability:  
  
**SQLite databases are first-class creation artifacts.**  
  
Databases exist alongside other artifact types such as:  
	•	images  
	•	videos  
	•	text outputs  
	•	latent tensors  
  
They are tracked in the archive and can be shared like any other creation artifact.  
  
⸻  
  
**Automatic Database Creation**  
  
Creations that read or write database outputs follow simple rules.  
  
If no database exists:  
	•	Beast2 automatically creates one.  
  
If a database already exists:  
	•	Beast2 appends new data to it.  
  
This allows knowledge to accumulate continuously with no manual setup.  
  
⸻  
  
**Deterministic Collaboration**  
  
Databases are designed to be **shared and merged between creators**.  
  
Example workflow:  
	1.	Creator A runs the research assistant creation.  
	2.	The system records all queries, inputs, and outputs.  
	3.	Creator B runs the same creation template locally.  
	4.	Both creators share their databases.  
	5.	The databases merge.  
  
Now both creators have access to the combined results.  
  
Because Beast2 records **creation DNA and deterministic parameters**, results can be compared and reproduced reliably.  
  
This transforms research into **collaborative distributed intelligence building**.  
  
⸻  
  
**Intelligent Caching**  
  
This creation uses **exact query caching**.  
  
When a query is executed:  
	•	the exact search term  
	•	search results  
	•	extracted content  
	•	LLM interpretation  
  
are stored in the database.  
  
If the same query appears again:  
  
Beast2 performs a **database index lookup instead of recomputation**.  
  
This returns results **instantly**.  
  
⸻  
  
**Emergent Encyclopedia**  
  
As many creators contribute queries and results, the database evolves into a structured knowledge resource.  
  
Over time it becomes:  
	•	a dictionary  
	•	a FAQ  
	•	an encyclopedia  
  
Because results are cached, answering previously asked questions requires **no additional compute**.  
  
This allows knowledge retrieval at **near-instant speed**.  
  
⸻  
  
**Compressed Compute Artifacts**  
  
LLM research results represent **expensive computation**.  
  
By storing summaries and interpretations, Beast2 converts expensive compute into **compact reusable artifacts**.  
  
This effectively compresses large amounts of computation into small knowledge entries.  
  
These artifacts can be shared and reused across creators.

**Research Algorithm**  
  
**1. Query Interface**  
  
The UI displays a prompt:  
  
**“Search for and interpret data while assuming it is related to Beast2.”**  
  
The system also injects Beast2 terminology definitions into the context.  
  
⸻  
  
**2. Multi-Engine Search**  
  
The query is sent to **three independent search engines**.  
  
Each engine returns **two results**, producing:  
  
6 search results  
  
Using multiple search engines improves result diversity.  
  
⸻  
  
**3. Content Extraction**  
  
Web pages often contain irrelevant HTML.  
  
A processing step extracts:  
	•	readable article text  
	•	headings  
	•	paragraphs  
	•	code blocks  
  
Navigation elements, scripts, and advertisements are removed.  
  
⸻  
  
**4. Context Assembly**  
  
The system prepares the LLM input using:  
	•	user query  
	•	Beast2 terminology definitions  
	•	extracted article text  
	•	source metadata  
  
This creates a structured context for interpretation.  
  
⸻  
  
**5. LLM Interpretation**  
  
The LLM generates:  
	•	a readable summary  
	•	insights related to Beast2 design  
	•	relevant technical explanations  
  
The output focuses on **interpreting research through the Beast2 architectural lens**.  
  
⸻  
  
**6. Database Storage**  
  
The creation stores all results in the SQLite database artifact.  
  
Typical stored fields include:  
	•	query text  
	•	search results  
	•	extracted article text  
	•	LLM interpretation  
	•	timestamps  
	•	creation template version  
	•	system parameters  
  
These entries become permanent archive artifacts.

Typical inputs include:
