	•	they create **shared expectations**  
	•	they reduce **accidental failure**  
  
However, your intuition is still correct in one sense:

That’s basically a core lesson of **computer security**.  
  
Security thinking assumes:  
  
```
rules exist
attackers search for edge cases
systems fail at boundaries

```
  
So engineers don’t aim for perfect rules. They aim for:  
  
```
defense in depth
multiple checks
fail-safe defaults
visibility

```

**3. Obfuscated code vs safe code**  
  
Your question:

Short answer: **neither axiom is correct**.  
  
There are three common cases.  
  
**1. Malicious concealment**  
  
Attackers hide behavior.  
  
Example:  
  
```
encoded strings
strange variable names
control flow tricks

```
  
Goal: make human review difficult.  
  
⸻  
  
**2. Intellectual-property protection**  
  
Some companies intentionally obfuscate code to prevent copying.  
  
Not malicious, but still **hard to audit**.  
  
⸻  
  
**3. Accidental obfuscation**  
  
Most real-world code isn’t intentionally hidden.  
  
It just becomes messy because:  
  
```
poor naming
no documentation
too many layers
legacy patches

```
  
This is actually the **most common case**.  
  
⸻  
  
**4. The real security axiom**  
  
A better rule engineers use is:  
  
```
Unreadable code cannot be trusted.

```
  
Not necessarily evil.  
  
But **high risk**.  
  
Because:  
  
```
bugs hide easily
security flaws hide easily
intent is unclear

```
  
So many projects adopt:  
  
```
readability > cleverness

```

At the deepest level a CPU performs operations like:  
  
```
input A

operation
output C

```

**Scotty Engineering Report: What Actually Makes Modern CPUs Weird**  
  
The part that surprises most software devs is that **the math is not the hard part anymore**.  
  
The real complexity is **moving data around fast enough**.  
  
The hierarchy looks like this:  
  
```
Registers (fastest)
↓
L1 cache
↓
L2 cache
↓
L3 cache
↓
RAM
↓
SSD

```
  
Typical latencies (very rough):  
  
```
Register: ~1 cycle
L1 cache: ~4 cycles
L2 cache: ~12 cycles
L3 cache: ~40 cycles
RAM: ~100–200 cycles
SSD: millions of cycles

```
  
So a CPU often spends most of its time **waiting for data**.  
  
⸻  
  
**Why Cache Exists**  
  
Caches store **recently used memory blocks**.  
  
They rely on two statistical truths:  
  
**Temporal locality**  
  
If you accessed something recently, you’ll likely access it again soon.  
  
**Spatial locality**  
  
If you accessed memory at address X, you might access X+1, X+2 soon.  
  
So caches move **blocks** instead of single values.  
  
Example:  
  
```
CPU asks for 4 bytes

```
Cache loads 64-byte line
