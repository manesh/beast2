**beast2-ui-ux-vocabulary-creations**  
  
**Introduction**  
  
**Beast2 is an infinite creation powertool.**  
Everyone is a beast. Everyone is a creator.  
Every creation becomes part of the archive.  
  
Beast2 is designed to make AI generation **fast, transparent, reproducible, and composable**.  
  
The system is built around a single concept: **creation**.  
  
A creation represents both the **instructions** and the **results** required to produce an artifact such as an image, video, or text output.  
  
Every creation is preserved in the archive so it can be:  
	•	inspected  
	•	modified  
	•	reproduced  
	•	extended  
	•	shared  
  
Nothing is lost. Even failures are valuable data.  
  
⸻  
  
**Core Vocabulary**  
  
**Creator**  
  
A **creator** is anyone who uses Beast2.  
  
There are no separate roles such as:  
	•	developer  
	•	artist  
	•	user  
  
Everyone interacting with Beast2 is a **creator**.  
  
Creators build, modify, and execute creations.  
  
⸻  
  
**Creation**  
  
A **creation** is the fundamental unit in Beast2.  
  
All creations are defined using the **b2 DSL**.  
  
The DSL is intentionally designed to behave more like a **template system** than a traditional programming language.  
  
Its goals are:  
	•	minimum keystrokes when editing text  
	•	minimum clicks in the UI  
	•	extremely fast creation of variants  
  
This solves a major pain point in existing AI tools where making small variations requires excessive manual work.  
  
⸻  
  
**Creation States**  
  
Every creation exists in one of three states.  
  
**Template**  
  
A **template** is a creation definition that has not yet been executed.  
  
It describes:  
	•	inputs  
	•	parameters  
	•	models  
	•	snippets  
	•	pipeline steps  
  
Templates are reusable and editable.  
  
⸻  
  
**Queued**  
  
A **queued** creation is scheduled for generation.  
  
Queued creations are waiting for GPU/CPU resources to execute.  
  
⸻  
  
**Complete**  
  
A **complete** creation has finished execution.  
  
At this point the system stores:  
	•	generated artifacts (images, video, text, latents)  
	•	parameters used  
	•	input references  
	•	pipeline steps  
	•	lineage relationships  
  
All of this information becomes linked to the creation template in the archive.  
  
⸻  
  
**Creation DSL**  
  
All creations are defined using the **b2 DSL**.  
  
Key design principles:  
	•	strict structure  
	•	safe for automated editing  
	•	optimized for rapid variant creation  
  
The DSL is designed to allow:  
	•	fast manual editing  
	•	reliable automated modification by local LLMs  
	•	strict validation to prevent errors  
  
This prevents “runaway edits” from breaking the system.  
  
⸻  
  
**Reproducibility**  
  
Conceptually, a Beast2 creation contains **everything required to reproduce an artifact exactly**.  
  
When exported, a creation becomes a .b2c file.  
  
A .b2c creation encapsulates:  
	•	all parameters  
	•	all pipeline steps  
	•	all input files  
	•	seeds  
	•	model references  
	•	intermediate outputs if needed  
  
Running the .b2c on another machine will reproduce the same result.  
  
⸻  
  
**Lineage**  
  
Every output generated by Beast2 contains lineage data.  
  
Clicking an output in the gallery shows:  
	•	which creation produced it  
	•	which inputs were used  
	•	which parameters were set  
	•	which upstream creations generated those inputs  
  
This makes the entire creation process transparent and reproducible.  
  
⸻  
  
**Archive**  
  
The **archive** stores all creations and their results.  
  
The archive grows continuously as creators work.  
  
Even unsuccessful outputs are preserved.  
  
This allows creators to:  
	•	analyze failures  
	•	compare parameter changes  
	•	rediscover useful ideas later  
  
⸻  
  
**Gallery**  
  
The gallery displays artifacts created by Beast2.  
  
Artifacts include:  
	•	images  
	•	videos  
	•	text outputs  
	•	intermediate results  
  
The gallery can also display external media files.  
  
Example workflow:  
  
Select an image from another program and use it as input for an **image-to-video creation template**.  
  
This allows Beast2 to integrate smoothly with other tools.  
  
⸻  
  
**Creation Types**  
  
**One-Shot**  
  
A **one-shot creation** produces a single output.  
  
Examples:  
	•	generate one image  
	•	generate one video  
	•	generate one text output  
  
One-shot creations are fast and simple.  
  
⸻  
  
**Gencast**  
  
A **gencast** generates multiple variants from a single template.  
  
Variants are created by modifying parameters or snippets.  
  
The goal is to **cast a wide net** to discover strong results.  
  
This is useful for:  
	•	prompt exploration  
	•	parameter tuning  
	•	discovering unexpected outputs  
  
⸻  
  
**Pipeline**  
  
A **pipeline creation** chains multiple operations.  
  
Example:  
  
creation → function(posterize) → creation → final output  
  
Pipelines may include multiple variant stages.  
  
Important design constraint:  
	•	pipelines are sequential  
	•	no branching trees  
  
This keeps the system predictable and easy to debug.  
  
⸻  
  
**Experimenter**  
  
An **experimenter creation** runs continuously or intermittently.  
  
It uses a **local LLM** to generate new creation variants.  
  
Safety mechanisms include:  
	•	strict DSL validation  
	•	no arbitrary code execution  
	•	template layer isolation  
  
Experimenter creations often store results directly in the database for later analysis.  
  
⸻  
  
**Summon the Beast**  
  
Beast2 includes a special feature in the help menu:  
  
**“Summon the Beast.”**  
  
This opens an interface to a local LLM trained on:  
	•	Beast2 documentation  
	•	design planning conversations  
	•	internal architecture notes  
  
The goal is to create **living documentation**.  
  
Creators can ask:  
	•	why the system works a certain way  
	•	how a feature was designed  
	•	how to modify the system  
  
Beast2 includes full source code and documentation so creators can easily modify it.  
  
⸻  
  
**Design Philosophy**  
  
Beast2 emphasizes:  
	•	transparency  
	•	reproducibility  
	•	rapid iteration  
	•	creator empowerment  
  
The system avoids hidden behavior and opaque pipelines.  
  
Everything is visible, inspectable, and modifiable.  
  
⸻  
  
**Batteries Included**  
  
Beast2 aims to be a **complete creative workstation**.  
  
The philosophy is:  
  
**“Batteries included, and the kitchen sink, and a helicopter.”**  
  
Creators should be able to:  
	•	generate assets  
	•	explore variants  
	•	build pipelines  
	•	run experiments  
	•	inspect lineage  
	•	reproduce results  
  
All within a single system.  
  
Beast2 is not just a generator.  
  
It is a **creation platform**.
